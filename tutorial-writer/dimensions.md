## 维度 1：动机引导

> "为什么我要学这个？"

**问题**：读者打开教程，第一段就被告知"今天我们学习 X"。还没有任何理由 care X 是什么，读到中间会走神，读完不会用。

**做法**：

- 用一个**具体场景**开场：读者会在什么情境下撞上这个概念？
- 制造一个**痛点**：没有这个东西时会出什么问题/有多麻烦？
- **然后**才介绍这个概念是来"解决这个痛点"的

**对照（数据库领域）**：

> ❌ 烂开头：外键约束是一种数据库完整性约束，它确保表与表之间的引用关系。

> ✅ 好开头：假设你做一个订单系统。你删掉了用户表里 ID = 7 的用户，但订单表里还有 5 条订单写着 `user_id = 7`。这些订单现在"挂在"一个不存在的用户上——查询会出错、统计会偏、迁移会爆。这种"孤儿数据"就是外键约束要解决的问题。

好开头**没有先定义概念**——它先让读者感受到痛。定义可以等到读者已经渴望解药再给。

**换个领域同样生效（Git）**：

> ✅ 你和同事都在改 `api.js`。他推了一个 commit 改了第 40 行，你本地也改了第 40 行。你 `git pull`，冲突。手动合并完，测试炸了——你发现他改的一个函数调用被你的版本覆盖了。这不是 Git 的 bug，是你不知道 merge 策略在做什么。Git 的合并策略就是来解决"两个人的修改怎么安全合在一起"这个问题的。

**换到 Python 异步**：

> ✅ 你写了个爬虫脚本，requests 一个一个发请求。500 个 URL 跑了 8 分钟。你打开 htop 一看——CPU 闲着、网卡闲着、整个进程就在等 socket。这就是同步阻塞的代价。`asyncio` 是来回答"等 IO 的时候能不能去干别的"这个问题的。

任何领域都一样：**先痛后药**。

## 维度 2：试错过程

> "先做做看 → 碰壁 → 理解"

**问题**：直接讲理论，读者会觉得"嗯好像懂了"，但合上书还是不会写。

**做法**：

- 让读者先尝试一个**朴素方案**（甚至是错的方案）
- 让方案在一个**具体输入**下真的失败
- 失败的现象引出"原来需要 X"——X 才是要教的东西

**例子（事务隔离级别）**：

1. "我们简单地两个会话同时改同一行试试"——给完整 SQL
2. 展示真实输出——出现脏读
3. "可以看到 A 还没提交，B 就读到了——这就是脏读。要避免它，需要……"

读者亲眼看到问题，后面讲的所有东西都有锚点。

**注意**：如果一个概念实在没法用一两行代码触发失败（太抽象），那么至少要给一个"如果没有这个概念，你会怎么写错"的反例。

## 维度 3：小节过渡

> "为什么这一节完了要读下一节"

**问题**：很多教程小节之间是"硬切"——上一节讲外键，下一节标题直接是"索引"，读者不知道两者关系。

**做法**：每节最后一句/段，**埋一个钩子**指向下一节。钩子的形式：

- 抛出新问题
- 揭示当前方案的局限
- 问"那如果……怎么办"

**例子**：

> 至此外键解决了"订单不能挂在不存在的用户上"。但你大概注意到了——每删一个用户，数据库要去订单表找一遍有没有引用。表大了之后这操作慢得吓人。所以下一节我们看：如何让"找一遍"足够快——索引登场。

下一节标题"索引"不再凭空冒出来。

## 维度 4：类比 / 判定表

> 替代规则罗列

**问题**：作者一气列出 5 条规则、6 个参数、7 种情况——读者眼睛扫过但脑子留不住。

**做法**：

- 多个并列概念 → **判定表**（什么情况下用哪个）
- 抽象概念 → **生活类比**锚定直觉
- 规则可以写，但**别只写规则**——后面接一个"也就是说，你可以把它想象成……"

**判定表例（ON DELETE 策略）**：

| 业务场景 | 选择 | 原因 |
|---------|------|------|
| 用户注销，订单也该一起没 | `CASCADE` | 主从关系，从属生命周期 |
| 用户注销，但历史订单要保留做财务对账 | `SET NULL` | 保留事实，断开引用 |
| 用户不能注销，除非先处理订单 | `RESTRICT` / `NO ACTION` | 强制业务流程 |

读者扫一眼就知道自己的场景该选哪个——比读 5 段文字快得多。

**类比例（事务的隔离级别）**：

> 隔离级别像图书馆的借阅规则。READ UNCOMMITTED 是"任何人随便翻你的草稿"；READ COMMITTED 是"只能看你交还回架的书"；REPEATABLE READ 是"你借这一本期间，别人改不了你这本"；SERIALIZABLE 是"全馆每次只让一个人进"——最安全但最慢。

## 维度 5：难点展开

> 别在容易混的地方一笔带过

**问题**：作者自己很熟，觉得"NO ACTION 和 RESTRICT 差不多"一句话带过。读者去用就踩坑。

**识别难点的信号**：

- 两个名字非常像的概念（`NO ACTION` vs `RESTRICT`、`==` vs `===`、过滤 vs 投影、`git reset` vs `git revert`）
- 行为看上去相同但语义不同
- 有多种"看上去等价"的写法
- 你自己第一次学时也搞混过——这是最强信号

**做法**：

1. **承认**"这两个看起来一样"
2. 给一个**两者行为分叉的最小例子**
3. 用判定表/对照表收束："实际中 xxx 用 A，xxx 用 B"

**展开 NO ACTION vs RESTRICT**：

> 表面上 `NO ACTION` 和 `RESTRICT` 都是"父行被引用就不让删"，但有个微妙差别——**检查时机**。
>
> - `RESTRICT`：检查在**这条 DELETE 语句执行的瞬间**就发生，引用还在就直接报错。
> - `NO ACTION`：检查推迟到**事务提交时**。这意味着你可以先 DELETE 父行，再 DELETE 子行，**只要事务提交前一切自洽**就行。
>
> 也就是说——如果需要在一个事务里"先删父再删子"的中间状态合法，就用 `NO ACTION`；如果想立刻报错防呆，就用 `RESTRICT`。

篇幅大概是普通段落的 2-3 倍，但这就是难点该有的待遇。**不要因为篇幅"看起来不平衡"而压缩难点**——平衡是节奏感骗局，读者卡在哪里就该在哪里花时间。

**同模式换到 Git**：

> 表面上 `git reset` 和 `git revert` 都是"撤销"，但有个根本差别——**是否改写历史**。
>
> - `git revert`：创建**一个新 commit**，内容是上一个 commit 的逆操作。历史是叠加的，适合已经 push 的公共分支——其他人 pull 到的是"多了一个撤销 commit"，不会断历史。
> - `git reset`：**移动分支指针**，直接丢弃 commit。适合还没 push 的本地改动——干净利落，但一旦已经 push 过，其他人 pull 时会冲突。
>
> 也就是说——commit 已经公开了就 `revert`，还在本地就 `reset`。**"是否已经 push" 是最简单的分界线**。

注意：同样是难点展开，但 Git 版的"分叉点"是**使用场景**（push 前/后），数据库版的"分叉点"是**行为机制**（检查时机）。识别每种难点的**真正分叉维度**，而不是套用固定模板。

## 维度 6：章节收束

> 常见误区 + 最佳实践

**问题**：读者读完一节，不知道"我学到的东西什么时候该用、什么时候不该用、容易在哪翻车"。

**做法**：每章/大节末尾收一个小段，包含：

- **常见误区**：新手容易做错什么（带反例）
- **最佳实践**：实际项目里怎么用是健康的
- 可选：**速查清单**（关键 API/参数 / 操作一行一个）

用 Obsidian callout 让它视觉上独立：

```markdown
> [!warning] 常见误区
> - 给所有外键都加 `ON DELETE CASCADE`——一旦误删一个用户，相关数据全没。
> - 在高并发表上用 `RESTRICT`——每次删除都要扫子表。

> [!tip] 最佳实践
> - 默认用 `RESTRICT`，需要级联时再显式改成 `CASCADE`。
> - 给外键列加索引（不少数据库不会自动加）。
```

**收束不是凑字数**——如果一节确实没有显著的误区，宁可只写"最佳实践"也别硬编。

## 维度 7：整体循序渐进

**问题**：第 5 节用到了第 8 节才讲的概念；或者第 2 节复杂度突然飙升。

**做法**：

- 写完后从头看一遍目录，问：**每一节用到的所有概念，是否都在前面讲过？**
- 如果某节必须用一个还没讲的概念，要么前移、要么明确写"这里先用一下，第 X 节展开"
- 复杂度**应该缓上升**——不要前 5 节都是 hello world，第 6 节直接上分布式事务
- 章节之间的**认知跨度**应该大致均匀，避免"忽快忽慢"

## 维度 8：参考完整度

> "读者能不能只靠这篇教程就把事做完？"

**问题**：教程的教学节奏完美——动机足、类比好、难点展开了——但读者读完仍然不会自己写代码。因为全文只有示例片段，没有完整的语法/API 定义、参数表、或跨平台差异说明。读者理解了概念，却需要去查官方文档才能写出第一行正确代码。

这是 1-7 维度的**集体盲区**——前面七个维度管"怎么讲"，第八个维度管"该讲的东西讲全了没有"。

**核验金句**：

> **读者合上这篇教程后，能不能完全不查外部资料就把这个功能独立用起来？**
>
> 能 → 参考完整度过关。不能 → 漏了参考块。

### 三种该上参考块的情况

**1. 有明确语法签名的核心概念，给出完整语法骨架框**（不只是示例）：

```sql
-- ✅ 完整语法框（读者可以对着它写任何变体）
CREATE [OR REPLACE] VIEW view_name [(column_list)]
AS query
[WITH [CASCADED | LOCAL] CHECK OPTION];

-- ❌ 只有示例（读者只能照抄，不能变通）
CREATE VIEW high_salary_employees AS
SELECT id, name, salary FROM employees WHERE salary > 80000;
```

骨架框的格式：**一行一个子句，可选部分用方括号标注，默认值标注在行内注释**。后面配一张"语法拆解表"逐项说明每个子句的含义、取值、默认值。

**2. 有多个参数/选项的配置项，给出参数表**：

| 参数 | 类型 | 默认值 | 说明 |
|:--|:--|:--|:--|
| `OR REPLACE` | 标志 | 不加 | 同名视图存在时原子替换 |
| `MATERIALIZED` | 标志 | 不加 | 创建物化视图（物理存储结果） |

**3. 涉及多平台/多实现时，给出兼容性矩阵**——不要用一句话带过差异：

```markdown
❌ 一句话带过（没信息量）：
"注意：不同数据库对 CHECK 约束的支持有差异。"
```

```markdown
✅ 兼容性矩阵（读者立刻知道自己的环境行不行）：
```

| 数据库 | CHECK 支持 | UDF in CHECK | 子查询 in CHECK |
|:--|:--|:--|:--|
| PostgreSQL | ✅ | ✅ IMMUTABLE | ❌ |
| MySQL 8.0+ | ✅ | 部分 | ❌ |
| SQLite | ✅ | ✅ | ⚠️ 不推荐 |

### 参考块的位置

放在"**解释**"之后、"**收束**"之前。读者先通过动机和试错建立直觉，再通过解释理解机制，然后参考块给出"现在你自己写的时候用这个骨架"——最后收束总结误区和实践。

### 判断参考块是否充分的三个问题

1. 读者读完这一节，能不能**不看任何外部资料**就写出正确的语法/API 调用？
2. 如果他用的数据库/浏览器/版本跟你的示例不同，文章有没有告诉他行为差异在哪？
3. 每个参数/选项的含义和默认值，文章有没有明确写出来（而不是仅靠示例暗示）？

三个都"是" → 参考完整度过关。

## 附录：课程风变体（写大学课程章节时叠加）

适用题目：操作系统、数据库系统、计算机网络、算法导论、高数、线性代数、离散数学、编译原理等系统课程的章节。

工程风默认骨架已经覆盖大部分场景，**课程风只在以下几个维度做叠加调整**——不是另起一套维度，是在原维度上"工整化 + 学术化"。

### 维度 1（动机）课程版

工程风的"实战痛点开场"依然要——但加一句**学术坐标**："在 X 课程体系里，这个概念是 Y 的核心齿轮"。让读者知道学这个不是孤立技巧，是课程链条的一环。

例：

> 假设你在内核里要实现进程切换。你需要在保存当前进程"全部状态"的同时，让 CPU 立刻去执行另一个进程——而且事后能完美恢复回来。这个"全部状态到底是什么、放在哪、怎么保存恢复" 就是上下文切换要解决的问题。**在操作系统课程里，这是进程管理（Ch3）连接调度算法（Ch4）的关键齿轮**——不理解上下文切换，后面所有调度算法你都看不懂为什么这么写。

### 维度 8（参考块）课程版

形式化定义要工整、术语要中英对照——这是教科书的传统：

```markdown
> [!abstract] 上下文切换（Context Switch）的形式化定义
> **定义**：操作系统将 CPU 控制权从一个进程（或线程）转移到另一个进程（或线程）的过程。
> 
> **包含的操作**：
> 1. 保存当前进程的执行状态（CPU 寄存器、PC、栈指针等）到其 PCB（Process Control Block）
> 2. 加载目标进程的状态从其 PCB 到 CPU 寄存器
> 
> **触发时机**：进程主动让出（yield）、时间片耗尽、I/O 阻塞、被更高优先级进程抢占。
```

工程风也可以这么写，但课程风**必须**这么写——这是教科书读者期望的表达方式。

### 章末必含：5-8 道混合习题

课程章节末尾必须有混合题（概念 + 应用 + 实操，混合配比），答案用折叠 callout。详细配比和模板见 [long-form.md](long-form.md) 的"章末综合（课程风）"节。
