# 教程写作与评审

写一篇好的技术教程，跟写参考文档不是一回事。参考文档的目标是"提供准确信息"，教程的目标是"让一个还不会的读者真的学会"。两者的写法差别巨大——但很多人下意识用文档思维写教程，导致读者读完仍然不会用。

这个技能围绕 7 个教学维度，帮你写出或修出一篇真正能教会人的教程。涉及 Obsidian 格式（wikilink、callout、frontmatter）时，配合 `obsidian-markdown` 技能。

## 何时使用

- 用户要写一篇关于某个概念/工具/技术的教程或指南
- 用户要评审一篇教程，问"这篇写得怎么样"或"按那 7 条标准过一遍"
- 用户要改进某个章节，比如"这节读起来跳，帮我改一下"
- 用户写技术博客、Obsidian 学习笔记、面向初学者的文档
- **不使用**："解释一下 X"但只想要几十字的口头回答；短问答不是长文教学，不触发本技能

## 两种工作模式

### 写作模式（从零写）

1. **先对齐目标读者**：他们已经会什么？卡在哪？为什么会来读这篇？
   - 如果用户没说，主动问一句——目标读者直接决定起点深度
2. **逆推目录**：每一节都要有教学意义，不是为完整性而存在
3. **每节按"动机 → 试错 → 解释 → 收束"四拍写**
4. **写完用评审清单自检一遍**再交付

### 评审模式（评审/改进已有教程）

1. **先读一遍全文，记整体感觉**——哪段读起来累、哪段觉得跳。这是直觉，比逐项打勾更准
2. **然后逐项过 7 维度**，每个维度给出**具体定位**：
   - 不要说"缺动机"，要说"第 1 节第 1 段直接定义了外键，没解释为什么读者会 care"
3. **区分两类问题**：
   - **结构性问题**（缺动机、跳跃、漏掉难点）→ 需要改写
   - **润色性问题**（措辞、例子不够鲜活）→ 顺手改
4. **给修改建议要有"改前 / 改后"对照**，不要光说"应该加点动机"

### 四拍与七维度的关系

写作模式的"四拍"是**节内节奏**，七个维度是**跨节质量属性**，不是一一对应：

| 四拍 | 对应维度 | 作用 |
|------|---------|------|
| 动机 | 维度 1（动机引导） | 每节开头回答"为什么读这一节" |
| 试错 | 维度 2（试错过程） | 朴素方案 → 碰壁 → 引出概念 |
| 解释 | 维度 4（类比/判定表）、维度 5（难点展开） | 用类比和展开替代平铺定义 |
| 收束 | 维度 6（章节收束） | 误区 + 最佳实践 |

三个维度**不入四拍、跨节生效**：
- **维度 3（过渡）**：在节尾埋钩子连接下一节
- **维度 7（循序渐进）**：管控整篇的前置依赖和复杂度曲线
- **维度 4 的判定表**可能跨多节复用

**不需要每节塞满 7 个维度**——短节可能只有动机+解释，难点节可能试错+展开占满整节。

## 输出骨架

拿到写作任务时，以下骨架作为默认起点，按目标长度选择。

### 短教程骨架（< 1000 字）

```markdown
## 为什么需要 [X]
[一个具体痛点场景，2-3 句]

## 先试试看
[最简可行代码，让读者跑起来]

## 发生了什么
[解释关键机制，配一个类比]

## 一个容易踩的坑
[最常见误区 + 正确做法]

## 收束
[什么时候用 / 什么时候不用的速查]
```

维度密度：开篇用动机 → 试错 → 类比 → 一个难点展开 → 收束。过渡和循序渐进由短篇幅自然保证。

### 长文骨架（> 2000 字）

```markdown
## 为什么需要 [X]
[详细痛点场景 + 没有 X 时的失败案例]

## 先试试看
[朴素方案 → 碰壁 → 现象分析]

## [概念 A]
[动机 → 试错 → 解释 → 收束，四拍完整]

## [概念 B]
[小节过渡钩子承接 → 四拍]

## [A vs B]：容易混的地方
[判定表 + 最小分叉示例，独立成节]

## [进阶概念 C]
[复杂度缓升，不跳跃]

## 常见误区与最佳实践
[全文收束：误区表 + 实践清单]
```

长文的每个主节走完整四拍，判定表和难点展开独立成节。章节数按需增减，原则是每节有教学意义、认知跨度均匀。

## 七个教学维度

下面每条先讲它要解决的真实问题，再给可操作的做法和反例。**理解 why 比记住 how 重要**——边界情况你才能自己判断。

### 维度 1：动机引导——"为什么我要学这个？"

**要解决的问题**：读者打开教程，第一段就被告知"今天我们学习 X"。还没有任何理由 care X 是什么，读到中间会走神，读完不会用。

**做法**：
- 用一个**具体场景**开场：读者会在什么情境下撞上这个概念？
- 制造一个**痛点**：没有这个东西时会出什么问题/有多麻烦？
- **然后**才介绍这个概念是来"解决这个痛点"的

**对照**：

> ❌ 烂开头：外键约束是一种数据库完整性约束，它确保表与表之间的引用关系。

> ✅ 好开头：假设你做一个订单系统。你删掉了用户表里 ID = 7 的用户，但订单表里还有 5 条订单写着 `user_id = 7`。这些订单现在"挂在"一个不存在的用户上——查询会出错、统计会偏、迁移会爆。这种"孤儿数据"就是外键约束要解决的问题。

好开头**没有先定义概念**——它先让读者感受到痛。定义可以等到读者已经渴望解药再给。

换个领域同样生效：

> ✅ 好开头（Git 版）：你和同事都在改 `api.js`。他推了一个 commit 改了第 40 行，你本地也改了第 40 行。你 `git pull`，冲突。手动合并完，测试炸了——你发现他改的一个函数调用被你的版本覆盖了。这不是 Git 的 bug，是你不知道 merge 策略在做什么。Git 的合并策略就是来解决"两个人的修改怎么安全合在一起"这个问题的。

数据库、Git、React 状态管理——任何领域都一样：先痛后药。

### 维度 2：试错过程——"先做做看 → 碰壁 → 理解"

**要解决的问题**：直接讲理论，读者会觉得"嗯好像懂了"，但合上书还是不会写。

**做法**：
- 让读者先尝试一个**朴素方案**（甚至是错的方案）
- 让方案在一个**具体输入**下真的失败
- 失败的现象引出"原来需要 X"——X 才是要教的东西

**例子（事务隔离级别）**：

1. "我们简单地两个会话同时改同一行试试"——给完整 SQL
2. 展示真实输出——出现脏读
3. "可以看到 A 还没提交，B 就读到了——这就是脏读。要避免它，需要……"

读者亲眼看到问题，后面讲的所有东西都有锚点。

**注意**：如果一个概念实在没法用一两行代码触发失败（太抽象），那么至少要给一个"如果没有这个概念，你会怎么写错"的反例。

### 维度 3：小节过渡——"为什么这一节完了要读下一节"

**要解决的问题**：很多教程小节之间是"硬切"——上一节讲外键，下一节标题直接是"索引"，读者不知道两者关系。

**做法**：每一节的最后一句/最后一段，**埋一个钩子**指向下一节。钩子的形式：
- 抛出新问题
- 揭示当前方案的局限
- 问"那如果……怎么办"

**例子**：
> 至此外键解决了"订单不能挂在不存在的用户上"。但你大概注意到了——每删一个用户，数据库要去订单表找一遍有没有引用。表大了之后这操作慢得吓人。所以下一节我们看：如何让"找一遍"足够快——索引登场。

下一节标题"索引"不再凭空冒出来。

### 维度 4：类比 / 判定表，替代规则罗列

**要解决的问题**：作者一气列出 5 条规则、6 个参数、7 种情况——读者眼睛扫过但脑子留不住。

**做法**：
- 多个并列概念 → **判定表**（什么情况下用哪个）
- 抽象概念 → **生活类比**锚定直觉
- 规则可以写，但**别只写规则**——规则后接一个"也就是说，你可以把它想象成……"

**判定表例子（ON DELETE 策略）**：

| 业务场景 | 选择 | 原因 |
|---------|------|------|
| 用户注销，订单也该一起没 | `CASCADE` | 主从关系，从属生命周期 |
| 用户注销，但历史订单要保留做财务对账 | `SET NULL` | 保留事实，断开引用 |
| 用户不能注销，除非先处理订单 | `RESTRICT` / `NO ACTION` | 强制业务流程 |

读者扫一眼就知道自己的场景该选哪个——比读 5 段文字快得多。

**类比例子（事务的隔离级别）**：
> 隔离级别像图书馆的借阅规则。READ UNCOMMITTED 是"任何人随便翻你的草稿"；READ COMMITTED 是"只能看你交还回架的书"；REPEATABLE READ 是"你借这一本期间，别人改不了你这本"；SERIALIZABLE 是"全馆每次只让一个人进"——最安全但最慢。

### 维度 5：难点展开——别在容易混的地方一笔带过

**要解决的问题**：作者自己很熟，觉得"NO ACTION 和 RESTRICT 差不多"一句话带过。读者去用就踩坑。

**识别难点的信号**：
- 两个名字非常像的概念（NO ACTION vs RESTRICT、`==` vs `===`、过滤 vs 投影、`git reset` vs `git revert`）
- 行为看上去相同但语义不同
- 有多种"看上去等价"的写法
- 你自己第一次学时也搞混过——这是最强信号

**做法**：
1. **承认**"这两个看起来一样"
2. 给一个**两者行为分叉的最小例子**
3. 用判定表/对照表收束："实际中 xxx 用 A，xxx 用 B"

**展开 NO ACTION vs RESTRICT**：

> 表面上 `NO ACTION` 和 `RESTRICT` 都是"父行被引用就不让删"，但有个微妙差别——**检查时机**。
>
> - `RESTRICT`：检查在**这条 DELETE 语句执行的瞬间**就发生，引用还在就直接报错。
> - `NO ACTION`：检查推迟到**事务提交时**。这意味着你可以先 DELETE 父行，再 DELETE 子行，**只要事务提交前一切自洽**就行。
>
> 也就是说——如果需要在一个事务里"先删父再删子"的中间状态合法，就用 `NO ACTION`；如果想立刻报错防呆，就用 `RESTRICT`。

篇幅大概是普通段落的 2-3 倍，但这就是难点该有的待遇。**不要因为篇幅"看起来不平衡"而压缩难点**——平衡是节奏感骗局，读者卡在哪里就该在哪里花时间。

同模式换到 Git：

**展开 git reset vs git revert**：

> 表面上 `git reset` 和 `git revert` 都是"撤销"，但有个根本差别——**是否改写历史**。
>
> - `git revert`：创建**一个新 commit**，内容是上一个 commit 的逆操作。历史是叠加的，适合已经 push 的公共分支——其他人 pull 到的是"多了一个撤销 commit"，不会断历史。
> - `git reset`：**移动分支指针**，直接丢弃 commit。适合还没 push 的本地改动——干净利落，但一旦已经 push 过，其他人 pull 时会冲突。
>
> 也就是说——commit 已经公开了就 `revert`，还在本地就 `reset`。**"是否已经 push"是最简单的分界线**。

注意：同样是难点展开，但 Git 版的"分叉点"是**使用场景**（push 前/后），数据库版的"分叉点"是**行为机制**（检查时机）。识别每种难点的**真正分叉维度**，而不是套用固定模板。

### 维度 6：章节收束——常见误区 + 最佳实践

**要解决的问题**：读者读完一节，不知道"我学到的东西什么时候该用、什么时候不该用、容易在哪翻车"。

**做法**：每一章/大节末尾收一个小段，包含：
- **常见误区**：新手容易做错什么（带反例）
- **最佳实践**：实际项目里怎么用是健康的
- 可选：**速查清单**（关键 API/参数 / 操作一行一个）

用 Obsidian callout 让它视觉上独立：

```markdown
> [!warning] 常见误区
> - 给所有外键都加 `ON DELETE CASCADE`——一旦误删一个用户，相关数据全没。
> - 在高并发表上用 `RESTRICT`——每次删除都要扫子表。

> [!tip] 最佳实践
> - 默认用 `RESTRICT`，需要级联时再显式改成 `CASCADE`。
> - 给外键列加索引（不少数据库不会自动加）。
```

**收束不是凑字数**——如果一节确实没有显著的误区，宁可只写"最佳实践"也别硬编。

### 维度 7：整体循序渐进

**要解决的问题**：第 5 节用到了第 8 节才讲的概念；或者第 2 节复杂度突然飙升。

**做法**：
- 写完后从头看一遍目录，问：**每一节用到的所有概念，是否都在前面讲过？**
- 如果某节必须用一个还没讲的概念，要么前移、要么明确写"这里先用一下，第 X 节展开"
- 复杂度**应该缓上升**——不要前 5 节都是 hello world，第 6 节直接上分布式事务
- 章节之间的**认知跨度**应该大致均匀，避免"忽快忽慢"

## 端到端示例：一篇微教程的完整节奏

以下是一篇约 400 字的微教程《git stash：临时"存档"你的工作》，展示 7 个维度如何在同一篇文章里协同：

---

### 为什么你需要 stash

你正在 feature 分支上改 `api.js`，改了一半，还没到能 commit 的程度。突然同事说 master 上有个紧急 hotfix 需要你立刻切过去 review。你不能 commit（半成品），也不能不管（改了一半的文件会跟着你切分支跑过去）。**你被卡住了**。`git stash` 就是为这个场景设计的——它把你的修改暂存到临时空间，还你一个干净的工作区。

### 先试试看

```bash
# 当前在 feature 分支，有未提交的修改
git status          # modified: api.js
git stash           # 暂存所有修改
git status          # working tree clean —— 干净了！
git checkout master # 自由切换
# ...review hotfix...
git checkout feature
git stash pop       # 恢复你的半成品
```

你看到 `api.js` 的修改回来了，和 stash 前一模一样。

### stash 到底是什么

可以把 stash 想象成一个**临时剪贴板栈**。每次 `git stash` 把当前修改"剪切"到栈顶，工作区恢复干净。`git stash pop` 把栈顶内容"粘贴"回来并从栈中移除。

### stash pop 和 stash apply 差在哪

这两个命令看起来都能恢复 stash——差别在**是否保留 stash 记录**：
- `pop`：恢复 + 删除记录（用后即弃）
- `apply`：恢复但保留记录（可反复应用到不同分支）

| 场景 | 用 |
|------|-----|
| 切回来继续原来的开发 | `pop` |
| 同一组修改想在多个分支上都试试 | `apply` |

### 常见踩坑

> [!warning] 别把 stash 当分支用
> stash 没有名字、没有历史。堆了七八个 stash 之后你根本分不清哪个是哪个。stash 是**临时**暂存——超过一天就该考虑开个分支。

> [!tip] 只 stash 部分文件
> `git stash push -p` 可以交互式选择哪些改动入 stash。

---

**这段例子里 7 维度的分布**：

| 段落 | 维度 |
|------|------|
| "为什么你需要 stash" | 维度 1（动机：具体痛点 + 被卡住的感觉） |
| "先试试看" | 维度 2（试错：可跑命令，读者立刻验证） |
| "stash 到底是什么" | 维度 4（类比：剪贴板栈） |
| "pop 和 apply 差在哪" | 维度 5（难点展开：最小分叉 + 判定表） |
| "常见踩坑" | 维度 6（收束：误区 + 最佳实践） |
| 各段衔接 | 维度 3（过渡：每段最后一句引出下一段） |
| 整篇顺序 | 维度 7（循序渐进：困境 → 基本用法 → 易混点 → 进阶技巧） |

注意：这篇不到 500 字，维度 3（过渡）和维度 7（循序渐进）由短篇幅自然保证，没有显式的"过渡钩子"段落——这就是短教程和长文在维度密度上的区别。

## Obsidian 输出格式约定

输出的教程应当是合法的 Obsidian markdown。

- **章节用 `##`，子小节用 `###`**——`#` 留给文章主标题（或者用 frontmatter 的 `title` 取代）
- **难点 / 提示用 callout**：`> [!info]`、`> [!warning]`、`> [!tip]`、`> [!example]`
- **代码块用三反引号 + 语言标识**（`sql`、`python`、`bash` 等）
- **关键术语首次出现时加粗**，后续出现可用 wikilink 关联：`[[外键约束]]`
- **frontmatter 加 tags**：例如 `#教程/数据库`

具体语法细节请配合 `obsidian-markdown` 技能。

## 写完自查的红旗（反模式）

- 第一段就开始定义术语 → **缺动机**
- 一节里出现 3 个以上"X 是 Y"句式 → **在堆定义而非教学**
- 出现 5+ 条平铺直叙的 bullet → **该上判定表**
- "简单来说，A 和 B 差不多"——后面没解释具体差别 → **难点漏展开**
- 章节末尾停在最后一个例子代码块 → **缺收束**
- 出现"我们之前提过 / 后面会讲"但实际上前面没提 / 后面没讲到位 → **结构断裂**
- 通篇没有过任何代码/SQL/命令 → **抽象到读者无法验证**

## 与 obsidian-markdown 技能的分工

| 关注点 | 技能 |
|--------|------|
| 文章**讲什么、怎么组织、教学节奏** | tutorial-writer（本技能） |
| 文章**怎么写成合法 Obsidian markdown** | obsidian-markdown |

写一篇 Obsidian 教程时两个技能通常一起用：tutorial-writer 决定内容结构，obsidian-markdown 提供语法。
