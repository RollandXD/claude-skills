# 教程写作与评审

技术教程跟参考文档不是一回事。**文档求准，教程求"读完真的会用"**。很多人下意识用文档思维写教程——先定义术语、再罗列规则、最后给代码——读者读完点头但不会写。

这个技能的核心是 **先痛后药、先错后理、先用后看**。围绕这条主线，提供两种工作模式（写作 / 评审）、九个分层教学维度、可复用的骨架和清单。

## 何时使用

- 用户要写一篇概念/工具/技术的教程或指南（"成稿"形式，无下限字数）
- 用户要评审一篇教程（"这篇怎么样"、"按维度过一遍"、"这节读起来跳"）
- 用户写技术博客、Obsidian 学习笔记、面向初学者的文档
- **不使用**：短问答、口头解释——这是对话不是教学

需要 Obsidian 语法（callout、wikilink、frontmatter）时，配合 `obsidian-markdown` 技能。

## 两种工作模式

### 写作模式

按题目复杂度分两条路（AI 自己判断，不必问用户）：

**简单题**（单概念教程、口语化标题如"理解 X"、"为什么需要 Y"）→ 直接写：

1. **对齐目标读者** — 他们已经会什么？卡在哪？为什么会来读？
   - **场景默认值**（用户没指明读者时按场景默认，不必每次都问）：
     - 公开博客 / 课程教程 → 假设读者是"刚学完前置知识但没实战过的同学"，必问起点
     - 团队内部材料 → 假设读者是"零基础队友"，起点拉到入门
     - Obsidian 个人笔记 → 假设读者是"6 个月后的自己"，可省略最基础解释、保留判断和坑点
2. **逆推目录** — 每节要有教学意义，不是为完整性而存在
3. **每节按"五段式"写** — 动机 → 试错 → 解释 → 参考块（按需）→ 收束
4. **写完按 A/B/C 三组扫维度自检** — 见 [review-checklist.md](review-checklist.md)
5. **末尾贴一份「读者复读简报」** — 用零基础读者视角扫一遍：

   ```
   ---
   ## 读者复读简报（自检）
   - 起点判断：[偏高 / 合适 / 偏低]，理由 ___
   - 跳跃感：第 N 节从 X 跳到 Y 中间少了 ___（无 → "无明显跳跃"）
   - 未解释术语：[读者第一次见到、文中没消化的词；无 → "无"]
   - 读者最可能卡住的地方：第 N 节 ___
   - 读者最可能想问的问题：___
   ---
   ```

   简报跑出来如有重大问题（起点偏、跳跃），先自己改了再交付。

**长文章节**（多小节章、题目带"操作系统 · 第 X 章"/ "数据库 · 索引与查询优化" 这种课程章节标识，或明确提到"教程书章节"）→ 启动 [long-form.md](long-form.md) 中的**两阶段工作流**：

1. **阶段 1：先输出大纲**（课程定位 + 节目录 + 学习目标 + 概念依赖图 + 术语表），等用户一次确认
2. **阶段 2：分批填充**，每节末尾贴自检标签（已覆盖维度 / 引入术语 / 难度评级）
3. **交付时附质量小报告**：维度覆盖 + 难度曲线 + 潜在风险 + **读者复读简报**

判定逻辑和模板都在 [long-form.md](long-form.md)，AI 写长文时必读。

每个维度的详细写法和对照例见 [dimensions.md](dimensions.md)。

### 评审模式

1. **先读全文记整体感觉** — 哪段读起来累、哪段觉得跳。直觉比逐项打勾更准
2. **按 A/B/C 三组过维度** — 每条给出**具体定位**："第 1 节第 1 段直接定义了外键，没解释读者为什么会 care"，而不是只说"缺动机"
3. **区分三类问题**：
   - **结构性**（缺动机、跳跃、漏难点、缺参考块）→ 需要改写
   - **风格性**（万能开场、套话过渡、列表病、不敢站队）→ 顺手改
   - **润色性**（措辞、例子不够鲜活）→ 顺手改
4. **建议带"改前 / 改后"对照** — 不要光说"应该加点动机"

复制粘贴的评审模板见 [review-checklist.md](review-checklist.md)。

## 九个教学维度（速查）

按"关注层面"分三组——AI 写每节时主要看 A 组，写完一章扫 B 组，全程在线 C 组。

### A. 节内教学节奏（写每一节都按此推进）

| # | 维度 | 一句话核心 |
|---|------|------|
| 1 | 动机引导 | 先制造痛点，再介绍解药 |
| 2 | 试错过程 | 让读者先用朴素方案碰壁，再讲机制 |
| 3 | 类比 / 判定表 | 多并列概念用判定表，抽象概念用生活类比 |
| 4 | 难点展开 | 易混概念别一笔带过，用最小分叉示例 |
| 5 | 章节收束 | 每节末尾给"误区 + 最佳实践" |

### B. 跨节连贯（写完一节、章后扫）

| # | 维度 | 一句话核心 |
|---|------|------|
| 6 | 小节过渡 | 节尾埋钩子，引出下一节 |
| 7 | 循序渐进 | 前置概念在前、复杂度缓上升 |

### C. 跨节元属性（贯穿全文的"质感"）

| # | 维度 | 一句话核心 |
|---|------|------|
| 8 | 参考完整度 | 读者能否合上教程独立写出正确代码 |
| 9 | 语气与风格 | 写出"Claude 人味"，不是"GPT 人机风" |

每个维度的**要解决的问题、做法、对照例**详见 [dimensions.md](dimensions.md)。

### 五段式与维度的对应

五段式是**节内节奏**，跟 A 组 1-5 严格一一对应：

| 节内段落 | 主对应维度（A 组） |
|---------|-------------------|
| 动机 | 1 |
| 试错 | 2 |
| 解释 | 3、4 |
| 收束 | 5 |
| 参考块 | 8（C 组元属性，按需嵌入） |

**跨节生效**：B 组的 6（节尾钩子）、7（整篇曲线）。
**全文在线**：C 组的 9（语气与风格）。

不需要每节塞满 9 个维度——短节可能只有动机+解释，难点节可能试错+展开占满。判断"参考块该不该有"的核验金句详见 [dimensions.md](dimensions.md) 维度 8。

## 输出骨架

| 目标长度 | 选哪套骨架 |
|---|---|
| < 2000 字 | 短教程骨架 |
| 2000–5000 字 | 长文骨架 |
| **> 5000 字 / 多小节章节 / 教科书一章** | **走 [long-form.md](long-form.md) 的两阶段工作流，不要直接闷头写** |

### 短教程骨架

```markdown
## 为什么需要 [X]
[一个具体痛点场景，2-3 句]

## 先试试看
[最简可行代码，让读者跑起来]

## 发生了什么
[解释关键机制，配一个类比]

## 一个容易踩的坑
[最常见误区 + 正确做法]

## 收束
[什么时候用 / 什么时候不用的速查]
```

短教程的"参考块"通常压缩为代码块中的注释（标注参数含义），不必独立成框。

### 长文骨架

```markdown
## 为什么需要 [X]                  详细痛点 + 没有 X 时的失败案例
## 先试试看                        朴素方案 → 碰壁 → 现象分析
## [概念 A]                        五段完整
## [概念 B]                        由钩子承接 A
## [A vs B]：容易混的地方           判定表 + 最小分叉
## [进阶概念 C]                    复杂度缓升
## 跨实现差异（如适用）             兼容性矩阵
## 常见误区与最佳实践               全文收束
```

如果不涉及多平台（纯理论概念），"跨实现差异"节可省略——别为对称硬凑。

一篇约 500 字的 git stash 微教程端到端示例（每段标注对应维度），以及"过评审清单发现漏参考块 → 修复 → 交付"的完整迭代过程，见 [examples.md](examples.md)。

## 与 obsidian-markdown 技能的分工

| 关注点 | 归谁管 |
|--------|------|
| 讲什么、怎么组织、教学节奏 | tutorial-writer（本技能） |
| 内容完整性——读者能否独立使用 | tutorial-writer（维度 8） |
| 语气与风格——写出人味 | tutorial-writer（维度 9） |
| 怎么写成合法 Obsidian markdown（callout 选哪个、frontmatter 字段、wikilink 形式） | obsidian-markdown |

分界线：**"要不要加语法定义框 / 参数表 / 兼容性矩阵" 是内容决策（本技能管），"用什么 callout 包它、表用几列" 是格式决策（obsidian-markdown 管）。**

写 Obsidian 教程时两个技能通常一起用——本技能定内容，obsidian-markdown 定语法。

## 写完自查的红旗（反模式）

输出有以下信号之一就回 [dimensions.md](dimensions.md) 对应维度修：

**A 组红旗（节内节奏）**

- 第一段就开始定义术语 → **缺动机**（维度 1）
- 一节里出现 3 个以上"X 是 Y"句式 → **在堆定义而非教学**（维度 2-4）
- 5+ 条平铺 bullet → **该上判定表**（维度 3）
- "A 和 B 差不多"后面没解释具体差别 → **难点漏展开**（维度 4）
- 章节末尾停在最后一个代码块 → **缺收束**（维度 5）
- 通篇无代码/命令/SQL → **抽象到读者无法验证**（维度 2）

**B 组红旗（跨节连贯）**

- "我们之前提过 / 后面会讲"但实际没提到位 → **结构断裂**（维度 6）
- 第 N 节用了第 N+M 节才讲的概念 → **前置概念后置**（维度 7）

**C 组红旗（元属性）**

完整性（维度 8）：
- 全文讲一个概念但没有完整语法/API 定义框 → **读者看完不会独立写**
- 全用单环境语法但教程面向多平台读者 → **换环境就踩坑**
- 关键参数的默认值 / 取值范围只靠示例暗示 → **关键信息埋藏**
- 出现"不同数据库对此支持不同"但没列差异 → **跨实现差异一句话带过**

风格（维度 9）：
- 第一段以"让我们 / 本文 / 在这篇教程中"打头 → **万能开场白**
- 全文超过 3 处"综上所述 / 值得注意的是 / 那么"打头的过渡 → **套话过渡**
- 一节里 3+ 个连续 bullet 列表（每条都很短） → **列表病**
- 全文找不到一句"X 比 Y 好 / X 更适合 / Y 是个坑" → **不敢站队**
- 出现"赋能 / 打造 / 一站式 / 丝滑 / 优雅地 / 深入探讨" → **AI 词典味**
