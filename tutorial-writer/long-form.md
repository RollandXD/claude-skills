# 长文章节专属流程

适用：单文件 5000+ 字、含多个相关概念的"章"级长文——典型如大学课程教科书一章（《操作系统 · 第 3 章 进程管理》、《数据库 · 索引与查询优化》），或工程领域的深度长文（《Git 全套必备技能》、《一万字搞懂 React 生命周期》）。

> **核心问题**：一次写 2 万字，用户没法逐字审。如果只在最后交付时一次性给 2000 行，用户已经"脱离对 AI 输出的掌控"——AI 中段的术语漂移、难度突跳、维度密度衰减都来不及拦截。
>
> **解决思路**：唯一一次重审放在**大纲阶段**（最便宜的拦截点），其余靠 AI 主动自检并把结果摊开给用户扫——节末标签 + 章末小报告。用户只读"标签 + 小报告"就能掌控全章。

## 两阶段工作流

### 阶段 1：先输出大纲，等用户确认

写章正文前**必须**先输出这份大纲。不要省略，不要合并到"我先写第一节"里。大纲必含 5 项：

```markdown
# [章标题] · 写作大纲

## 1. 课程定位（AI 自动推断）
[这章在课程体系里的位置]
例: "操作系统 · Ch3 进程管理 · 上承 Ch2 中断与陷入（前置：用户态/内核态切换），
     下接 Ch4 调度算法（本章给出'进程是什么'，下章决定'CPU 给谁'）"

## 2. 学完能做什么（教育目标，3-5 条）
- 看懂 `ps` 命令每一列对应的内核数据结构
- 解释为什么 fork() 后父子进程的栈是独立的
- ...

## 3. 节目录与每节学习目标
- 3.1 进程的概念：建立"进程 = 程序 + 上下文"的直觉，理解 PCB
- 3.2 进程状态转换：掌握五态模型与转换触发条件
- 3.3 上下文切换：理解保存/恢复机制与开销来源
- 3.4 进程间通信（IPC）：对比管道/消息队列/共享内存的适用场景
- 3.5 同步与互斥：信号量、互斥锁、条件变量的语义差异
- 3.6 经典问题：生产者-消费者、读者-写者、哲学家就餐

## 4. 概念依赖图（哪节用到哪节才讲的概念）
- 3.2 依赖 3.1 的 PCB
- 3.3 依赖 3.1 的 PCB、3.2 的进程状态
- 3.5 依赖 3.4 的"共享内存"概念
- 3.6 依赖 3.5 的全部同步原语
[识别"前置概念后置出现"问题——如果有，要么调整顺序，要么标注"先用一下"]

## 5. 术语表（本章会引入的所有新术语，写正文时严格按这个命名）
| 术语 | 中英对照 | 首次出现节 | 一句定义 |
|---|---|---|---|
| PCB | 进程控制块 (Process Control Block) | 3.1 | 内核中保存进程全部状态的数据结构 |
| 上下文切换 | Context Switch | 3.3 | OS 将 CPU 控制权从一个进程转移到另一个的过程 |
| ... | ... | ... | ... |

---
**大纲自检**:
✅ 课程定位明确 / 概念依赖无断裂 / 术语表无遗漏
⚠️ 风险提示: [例如 "3.4 IPC 涉及内核态知识，若读者未学过 Ch2 可能吃力"]

是否按此大纲展开？（不回复视为通过，5 秒后开始写正文）
```

### 阶段 2：分批填充正文

按节顺序输出。**每输出 2-3 节作为一批**——不要一次甩 6 节，也不要一节一节地慢慢挤。每节末尾贴自检标签：

```markdown
[正文...]

---
> [!abstract]- 本节自检
> ✅ 已覆盖维度: 1（动机）, 2（试错）, 4（类比 + 判定表）, 5（PCB vs TCB 难点展开）, 6（误区/最佳实践）, 8（PCB 字段速查表）
> ⚠️ 故意跳过: 3（本节为章首，无前节可衔接）
> 📌 本节引入术语: PCB, fork(), exec()
> 🎯 本节难度: ★★（基础）
> 🔗 引用前节术语: —（本节无）
```

折叠 callout 让用户能选择性展开——觉得这节没问题就折着；想抽查哪节就点开。

### 阶段 3：交付时附整体小报告

全章写完最后一节后，单独输出一份小报告（不嵌在正文里）。这份报告是用户审阅长文的**主要抓手**：

```markdown
## 写作小报告

**章标题**: 操作系统 · 第 3 章 进程管理
**总长度**: 约 18,400 字（含代码与表格）
**整体维度覆盖**: 8/8 ✅

### 难度曲线
3.1 ★★（基础）          ▰▰
3.2 ★★★（中）           ▰▰▰
3.3 ★★★★（中高）        ▰▰▰▰
3.4 ★★★★（中高）        ▰▰▰▰
3.5 ★★★★★（高）⚠️ 难点  ▰▰▰▰▰
3.6 ★★★（综合应用）      ▰▰▰

### 术语一致性自检
✅ 全章 14 个术语命名与大纲一致，未发现漂移

### 概念依赖自检
✅ 所有"用到的概念"都在前面节讲过，无断裂

### 潜在风险（自查发现）
- 3.4 → 3.5 难度跳了 1 级，但 3.5 是难点节符合"集中难度"模式 ✓
- 3.5 信号量小节较长（约 4200 字），如读者疲劳可拆分

### 建议你重点扫一眼这几节
1. **3.5 同步与互斥**——最难、最容易出问题的一节
2. **3.4 IPC**——如果读者无 Ch2 背景，开头可能要补一句

### 章末综合（按风格分流）
- 本章为课程风 → 已附 8 道混合题（2 概念 + 3 应用 + 3 实操），答案用折叠 callout
- 已附"本章术语速查表"
```

用户看完这份小报告 + 扫一遍每节自检标签，就能掌握全章质量——**不必逐字读 2 万字**。

## 章级结构（节内五段式之外的结构）

长文不是把 6 个独立小节堆起来——"章"作为整体要有自己的开头、收束、术语回顾。

### 章引导段（章首必备）

写在章标题之后、第一个小节之前。两个 callout：

```markdown
# 第 3 章 进程管理

> [!info] 本章定位
> 上承 Ch2 中断与陷入（你需要知道 OS 怎么从用户态切到内核态）；
> 下接 Ch4 调度算法（本章解决"进程是什么"，下章决定"CPU 给谁"）。
> 本章解决：进程是什么、怎么创建、怎么切换、怎么协作。

> [!example] 学完你能做什么
> - 看懂 `ps` 命令每一列代表的内核数据结构
> - 解释为什么 `fork()` 之后父子进程的栈是独立的
> - 用信号量正确实现一个线程安全的有界缓冲区
> - 分析"哲学家就餐"问题为什么需要打破环路
```

工程风可以省掉"课程定位" callout，但"学完能做什么"必须有——给读者一个走完全程的承诺。

### 章末综合（章末必备，按风格分流）

**工程风**：踩坑清单 + 最佳实践速查 + 完整命令速查表

```markdown
## 本章踩坑速查

> [!warning] 常见误区
> - 给所有外键加 CASCADE → 误删一人，关联数据全没
> - ...

> [!tip] 最佳实践
> - 默认 RESTRICT，需要级联时显式改 CASCADE
> - ...
```

**课程风**：5-8 道混合题（必备）

题型混合按 iv 配比：

- **概念题**（2-3 题）：考定义、机制、原理 — "解释一下进程和线程的本质区别"
- **应用题**（2-3 题）：给场景让读者分析 — "下面这段代码会不会死锁？为什么？"
- **实操题**（1-2 题）：给代码 / SQL / 命令让读者写或调试 — "用信号量实现 N 个生产者 M 个消费者的有界缓冲区"

答案用折叠 callout（默认折起，读者先想再展开）：

```markdown
## 章末练习

### 1. 概念题
解释"进程上下文切换"具体切了哪些东西？为什么开销大？

> [!note]- 答案
> 切换三类内容：
> 1. CPU 寄存器（通用 + PC + SP + 标志）
> 2. 内存映射（页表基址寄存器、TLB 失效）
> 3. 内核栈
> 开销主要来自 TLB 失效后的缓存重建，以及切换后冷启动的指令/数据 cache miss。

### 2. 应用题
某 OS 的进程切换耗时 5μs，时间片 1ms。如果系统有 100 个 CPU 密集型进程，CPU 利用率（用于真正计算）是多少？

> [!note]- 答案
> 每个时间片 1ms 内，4995μs 用于计算，5μs 用于切换。
> 利用率 = 1000 / 1005 ≈ 99.5%
> 如果时间片降到 50μs，利用率 = 50/55 ≈ 90.9% — 显著下降。
> 这说明时间片不能太小。

### 3. 实操题
[...]
```

### 章末术语回顾（课程风必备，工程风可选）

放在章末综合之前，让读者不必往前翻找术语：

```markdown
## 本章术语速查

> [!abstract]
> | 术语 | 定义 | 首次出现节 |
> |---|---|---|
> | PCB（进程控制块） | 内核中保存进程全部状态的数据结构 | 3.1 |
> | 上下文切换 | OS 将 CPU 控制权从一个进程转移到另一个的过程 | 3.3 |
> | 信号量 (Semaphore) | 用于进程间同步的整数计数器 + 等待队列 | 3.5 |
```

**与大纲阶段术语表的关系**：大纲那张是"要引入什么"的合同；章末这张是"已引入什么"的回顾。两者必须**字段一致**——AI 写完后做一次比对，不一致就是漂移。

## AI 自检协议（防漂移核心）

写长文最容易漂移的几个点，AI **必须主动管**——不是用户来挑刺，是 AI 写的过程中就消化掉。

### 1. 术语命名一致性
- **大纲阶段的术语表是合同**——后面填充必须按那个命名
- 反例：3.1 写 "PCB（进程控制块）"，3.5 写成"进程描述符" → **漂移，必须改回**
- 中英对照在大纲定下来，全章不变

### 2. 难度曲线监控
- 每节自评 ★ 数（1-5）
- 相邻节难度跳跃 ≤ 1 级（难点节可跳 2，但要在小报告里标注 ⚠️）
- 章末小报告画出曲线让用户扫

### 3. 维度密度不能后期下降
- 每节都要过 8 维度自检（标签里贴）
- **失败信号**：连续两节都"故意跳过维度 X" → 不是真的不需要，是写累了开始偷懒，回头补
- 中段（3-4 节）通常是最容易塌的——这里要刻意保住判定表/类比/难点展开

### 4. 前置概念检查
- 每节开头扫一遍依赖：本节用到的非通用术语，是否都在前面的节里讲过？
- 没讲过 → **立刻报警**，要么前移那节，要么写"这里先用一下，第 X 节展开"
- 自检标签里贴 "🔗 引用前节术语" 就是给自己做这个检查

### 5. 风格不漂移
- 选定工程风/课程风后，全章保持
- 反例：3.1-3.3 工程风口语化，3.4 突然教科书定义堆 → 风格断裂
- 章引导用什么风格、章末综合用什么形式，定了就贯彻

## 风格分流（工程 vs 课程）

| 维度 | 工程风（Git / 语言 / 工具） | 课程风（OS / DB / 高数 / 算法） |
|---|---|---|
| 章首引导 | 实战痛点 + "学完能做什么" | 课程定位（X 课 · Ch_N · 上下章关系）+ "学完能做什么" |
| 节内动机（维度 1） | 实战痛点开场 | 实战痛点 + 一句"在课程体系里这是 X 的核心齿轮" |
| 节标题风格 | 口语化（"为什么需要 rebase"） | 工整（"3.1 进程的概念"、"3.2 进程状态转换"） |
| 维度 8 参考块 | 命令速查 + 兼容性矩阵 | 形式化定义（中英对照）+ 触发时机/前置条件枚举 |
| 章末综合 | 踩坑清单 + 最佳实践速查 | 5-8 道混合题（概念+应用+实操，答案折叠） |
| 章末术语回顾 | 可选 | **必备** |

**判定规则**（AI 自动判断，不必问用户）：

- 题目含"操作系统"、"数据库系统"、"计算机网络"、"算法导论"、"高数"、"线性代数"、"离散数学"、"编译原理"、"X 第 N 章" → **课程风**
- 题目含 "Git"、"React"、"如何用 X"、"X 入门"、"X 实战"、"X 必备技能" → **工程风**
- 同时出现 / 不明确（如"Python 进阶"）→ **默认工程风 + 加章末练习题**（取两家之长）

## 长文章节常见反模式

写完后过一遍这个清单，命中任一条就回去修：

- 章引导只写"本章学 X" → **没有定位 + 没说能做什么**
- 章首跳过引导段直接进 3.1 → **章作为整体不存在**
- 6 个小节风格忽工程忽课程 → **风格漂移**
- 章末就是最后一节的代码块 → **缺章末综合**
- 第 4 节用了第 6 节才讲的术语 → **概念依赖断裂**
- 中段 3 节都没用判定表 / 类比 / 难点展开，全是定义堆 → **维度密度后期下降**
- 术语在 3.1 叫 PCB，到 3.5 写成"进程描述符" → **命名漂移**
- 课程风但章末没有练习题 → **失去课程风的核心特征**
- 工程风但章末突然出 8 道考试题 → **风格不一致**
- 大纲阶段术语表 14 个，章末术语回顾只有 9 个 → **5 个术语凭空消失或临时新增**
