# 评审清单（评审模式直接照打）

直接复制下面这块作为输出模板。**每条都要给具体定位**（章节标题 / 段落 / 行号）和**改前 / 改后对照**——光打勾或泛泛而谈"应该加点动机"不算评审。

```markdown
## 教程评审

### A. 节内教学节奏

#### 1. 动机引导
- [ ] 开头是否给了具体场景/痛点？
- 定位与建议：

#### 2. 试错过程
- [ ] 是否让读者先尝试朴素方案再引出概念？
- 定位与建议：

#### 3. 类比 / 判定表
- [ ] 罗列规则的地方是否配了判定表或类比？
- 定位与建议：

#### 4. 难点展开
- [ ] 容易混的概念是否充分展开（不一笔带过）？
- 定位与建议：

#### 5. 章节收束
- [ ] 每章末尾是否有常见误区 / 最佳实践？
- 定位与建议：

### B. 跨节连贯

#### 6. 小节过渡
- [ ] 各节之间是否有明确的"钩子"承接？
- 定位与建议：

#### 7. 循序渐进
- [ ] 是否有"前置概念后置出现"或"复杂度突跳"？
- 定位与建议：

### C. 跨节元属性

#### 8. 参考完整度
- [ ] 每个核心语法/API 是否有完整的语法骨架框（不只是示例）？
- [ ] 有多个参数/选项时是否有参数表（参数名、类型、默认值、说明）？
- [ ] 涉及多平台/多实现时是否有兼容性矩阵（不只是一句话带过）？
- [ ] 读者能否只靠本文（不查外部文档）独立写出正确代码？
- 定位与建议：

#### 9. 语气与风格
- [ ] 是否有万能开场白（"让我们 / 本文将 / 在这篇教程中"）？
- [ ] 是否有套话式过渡（"综上所述 / 值得注意的是 / 那么"打头）？
- [ ] 列表 / 加粗是否过度（什么都拆 bullet、什么都加粗）？
- [ ] 是否敢站队（出现"X 比 Y 好" / "Y 是个坑" / "官方文档讲反了"类判断）？
- [ ] 是否有 AI 词典味（"赋能 / 打造 / 一站式 / 丝滑 / 优雅地 / 深入探讨"）？
- 定位与建议：

## 总评

**结构性问题**（要改写，A、B 组 + 维度 8）：

**风格性问题**（顺手改，维度 9）：

**润色性问题**（措辞、例子鲜活度等）：
```

## 使用提示

- 先读全文记整体感觉再过清单——直觉先于打勾
- "定位"必须精确到章节标题/具体段落，不是"前面那部分"
- "建议"要可执行：给出具体的改前/改后文字，不要只说"加强动机"
- 总评分三类——结构性的需要改写、风格性的顺手改、润色性的顺手改，不要混在一起报
- 风格问题（维度 9）通常**面广量大但单点改动小**——不要一条一条单独列"第 X 段万能开场白"，可以归并成"全文 4 处万能开场，分别在 ___，统一改为直接进场景"
